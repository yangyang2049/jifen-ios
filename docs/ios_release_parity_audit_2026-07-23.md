## 1. 发布结论

| 判定 | 结果 |

| 代码内已知 P0 | **0** |

当前不能直接写成“可上线”，原因不是仍有已知崩溃，而是还有一个外部 P1 和两类必须由真实环境完成的发布门禁：

1. **P1：官网条款/隐私说明与本次 iOS 实际能力不一致。** 当前 iOS 2.0 是本地功能和 Apple Watch 联动版本，没有账号登录、会员/IAP、云同步或 WebSocket 展示端；官网条款仍写有 iOS 微信登录、账号注销、云端与在线同步等能力。必须先修正文案，或真正交付对应能力，并同步核对 App Store Connect 的功能描述与隐私标签。
2. **真实 iPhone + Apple Watch 联动门禁尚未完成。** 模拟器与自动化无法替代蓝牙/Wi‑Fi 切换、锁屏、后台、杀进程、断网完赛后恢复等真实链路。
3. **商店侧门禁尚未完成。** 当前产物为 `2.0 (1)`，仍需确认 App Store Connect 中 build 号唯一，使用正式签名 Archive，上传后完成 TestFlight/审核前检查。

Apple 要求提交前测试崩溃和缺陷、保证元数据准确，并为审核提供完整访问与说明，见 [App Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)；2026 年当前上传 SDK 要求见 [Submitting to the App Store](https://developer.apple.com/app-store/submitting/)。

### 2.4 有意保留的范围差异

- 账号、会员、IAP、云端积分同步、展示端与反馈仍未纳入本次 iOS 2.0。它们是明确的产品范围差异，不是把未实现功能伪装成“已对齐”；官网与商店文案必须按实际范围收敛。
- iOS 使用 `WatchConnectivity` 的可达消息、后台传输和本地持久化队列，不照搬 HarmonyOS P2P 的心跳频率。两端应对齐的是“唯一主控、幂等、可重试、断连不丢结果”，而不是底层传输形态。
- iOS 保留 Apple 平台的导航、触感、动态字体、横竖屏与系统权限体验，不要求视觉结构机械复制 HarmonyOS/Android。

## 3. Apple Watch 对照

### 3.1 独立功能

Apple Watch 当前与 HarmonyOS 手表基线一致：

- 独立计分 12 个模式：乒乓球、羽毛球、网球、匹克球的单打/双打，射箭、黑八、九球追分、斯诺克。
- 独立训练：投篮训练。
- 本地计时、工具与本地记录闭环。
- 篮球正式比赛入口不在手表首页，避免把保留代码误当已发布功能。

### 3.2 手机发起联动范围

手机发起联动覆盖：

- 乒乓球：单打、双打
- 羽毛球：单打、双打
- 网球：单打、双打
- 匹克球：单打、双打
- 射箭
- 黑八
- 九球追分（2–4 人）
- 斯诺克

投篮训练属于手表独立训练，不进入手机发起比赛联动。当前范围与 HarmonyOS 实际发布基线一致。

### 3.3 本轮修复的联动 P1

| 问题 | 风险 | 修复 |
| --- | --- | --- |
| 重试快照被判重复后不回 ACK | 发送方持续等待，联动假卡死 | 手机与手表均区分“新修订/重复/过期”，重复消息也幂等回 ACK |
| 手机保存完赛记录失败仍向手表 ACK | 手表清掉待发记录，比赛记录永久丢失 | 完赛落库改为可抛错；只有持久化成功才 ACK，失败保留重试 |
| 手表独立完赛在 WCSession 未激活时直接发送 | 未配对/暂不可达时记录可能丢失 | 增加持久化 outbox，激活和可达恢复后自动冲刷 |
| 手机接管未等手表确认就成为主控 | 两端可能同时操作，产生双主控 | 改为 ACK 握手；手表确认后手机才完成角色切换 |
| 重连后两端角色可能漂移 | UI 与实际主控不一致 | 手机在可达恢复时查询手表状态，并按手表权威角色恢复本地角色 |
| 手表遇到协议版本不匹配只静默超时 | 用户只看到等待，无法判断原因 | 明确返回 setup rejected，由手机展示失败 |

Apple 侧的增强点是：消息处理幂等、完赛先落库后确认、Watch 独立记录持久化 outbox、系统离线传输，以及接管 ACK 交接。这些能力比简单复制鸿蒙协议字段更符合 Apple 平台，也直接覆盖了结果不丢和唯一主控两个最高风险点。

## 4. 崩溃、数据与体验复检

本轮同时处理：

- 移除多处设置对象强制解包，避免恢复状态或异常导航时触发崩溃。
- 移除掼蛋、专项计分板和 Watch 射箭路径中的风险强制解包。
- 修复资源目录中 AppLogo 2x/3x 的异常文件名，消除资源编译告警。
- 清理已废弃的 `WKProcessPool` 使用和未消费返回值告警。
- 中英文 App 文案键精确对齐为 915 / 915，Watch 为 194 / 194；重复键为 0。
- 删除未实现处理器的 `jifen://` URL Scheme，避免商店产物声明不可用入口。
- 修正 LaunchScreen 构建配置，Release 产物不再出现嵌套 `UILaunchScreen`。
- 修复篮球计分板旋转瞬间中心栏宽于可用宽度而产生负 frame 的问题，并对设置弹窗瞬态宽度做非负夹取；6 个重构计分板横屏回归已确认无 `Invalid frame dimension` 运行时告警。
- iPhone 与 Watch 均增加 Privacy Manifest，声明 UserDefaults 必需原因 `CA92.1`、不跟踪、无采集数据。Apple 的配置说明见 [Adding a privacy manifest](https://developer.apple.com/documentation/bundleresources/adding-a-privacy-manifest-to-your-app-or-third-party-sdk) 与 [Required reason API](https://developer.apple.com/documentation/bundleresources/describing-use-of-required-reason-api)。

全盘扫描未发现 `fatalError`、`try!`、`as!` 等明显发布崩溃入口；剩余的 `!` 为系统代理签名或经硬编码保证有效的 URL。记录存储使用原子写入；存储空间耗尽时统一向用户展示失败和恢复动作仍属于 P2 韧性增强。

## 6. 上线前必须完成

### P1 / 阻塞

- 修正 [官网服务条款](https://www.jifenqi.com/terms) 与 [隐私政策](https://www.jifenqi.com/privacy) 中超出本次 iOS 实际能力的账号、支付、云同步和在线能力描述；同步复核 App Store Connect 元数据与隐私标签。
- 用真实 iPhone + Apple Watch 覆盖：手机发起所有支持项目、双方增减分、断连重连、锁屏/后台、手表离线完赛后回传、接管时切网、双方杀进程后恢复、记录只落一份且不重复。

### 商店发布门禁

- 将 build 号调整为 App Store Connect 中未使用的值。
- 使用 Distribution 签名 Archive，验证嵌套 Watch App、隐私清单、图标与 dSYM 后上传。
- TestFlight 新安装、升级安装各做一轮；复核相册权限、横竖屏、深浅色、中文/英文、动态字体和 VoiceOver 关键入口。
- 审核说明明确：本版本本地使用、Apple Watch 联动范围、无登录/无云同步；提供可直接复现的联动步骤。

### P2 / 后续增强

- 对存储耗尽/写入失败统一提供用户可见错误、重试与导出恢复。
- 增加足球、拳击、通用台球、简易计分和多人计分板的专项边界自动化。
- 做“手机与手表均被杀进程后继续实时比赛”的显式会话恢复；当前完赛 outbox 能保证结果不丢，但不等同完整实时会话续接。
- 结合实际用户系统分布重新评估 iOS 17.6 最低版本，而不是仅以构建通过作为产品决策。
